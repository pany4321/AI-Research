## 1. 监督微调（SFT）

### 1.1 原理

监督微调（Supervised Fine-Tuning）是后训练的起点。在预训练模型的基础上，使用高质量的指令-回答对进行标准的下一个 token 预测训练，使模型学会遵循指令、格式化输出。

**公式**：
$$
\mathcal{L}_{\text{SFT}} = -\mathbb{E}_{(x,y) \sim \mathcal{D}} \left[ \sum_{t=1}^{|y|} \log \pi_\theta(y_t | x, y_{<t}) \right]
$$

其中 $x$ 为指令，$y$ 为期望的输出，$\pi_\theta$ 为策略模型。

### 1.2 数据构建

#### 数据来源

| 来源类型 | 方法 | 代表示例 |
|---------|------|---------|
| 人工标注 | 雇佣标注员按 guidelines 编写 | InstructGPT 的 12.7K 演示数据 |
| 开源数据集 | 社区积累的指令数据 | OpenAssistant, ShareGPT, WildChat |
| 模型合成 | 强模型生成 + 过滤 | Alpaca (Self-Instruct), Dolly |
| 拒绝采样 | 采样多个回答 → 筛选最高分 | Llama 3.1 的 Rejection Sampling (K=10~30) |

#### 数据质量策略（Llama 3.1 实践）

1. **多维度评分**：主题分类、质量评分（1~5 分）、难度评分、语义去重
2. **差异化训练**：高质量数据多 epoch 训练，低质量数据降采样
3. **去重**：MinHash + LSH 去重，防止记忆效应
4. **数据配比**：按领域配比采样（数学、代码、写作、对话等），避免灾难性遗忘

### 1.3 训练配置与调优

#### 关键参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 学习率 | 5e-6 ~ 2e-5 | 全参比 LoRA 小 1~2 个数量级 |
| 训练轮数 | 1~3 epoch | 过多导致过拟合，偏好数据尤其注意 |
| 序列长度 | 模型最大 ctx 的 80%~100% | 长序列需配合 FlashAttention-2 |
| Batch Size | 尽可能大（梯度累积） | 影响收敛稳定性 |
| 权重衰减 | 0.01 ~ 0.1 | 防过拟合 |
| 梯度裁剪 | max_norm=1.0 | 防梯度爆炸 |

#### LoRA 配置

```yaml
# LLaMA-Factory 配置示例
finetuning_type: lora
lora_rank: 8
lora_alpha: 16
lora_dropout: 0.05
lora_target: all          # 或指定为 q_proj,v_proj
```

#### 2025 年趋势

- **Curriculum SFT**：先训基础能力（短回答），再训长思维链（Long CoT），Light-R1 验证有效
- **Neat Packing**：防止跨样本污染的 packing 策略，兼顾效率和质量
- **On-policy SFT**：使用当前模型采样数据来训练自身，减少分布偏移

### 1.4 TRL 代码示例


## 2. 奖励建模（Reward Modeling）

### 2.1 原理

奖励模型（Reward Model, RM）的目标是学习人类偏好。给定同一个 prompt 的多个回答，RM 预测人类评分员对每个回答的偏好分数。它是 RLHF 流程中 PPO 阶段的奖励信号来源。

**Bradley-Terry 模型**：
$$
\mathcal{L}_{\text{RM}} = -\mathbb{E}_{(x, y_w, y_l) \sim \mathcal{D}} \left[ \log \sigma \left( r_\phi(x, y_w) - r_\phi(x, y_l) \right) \right]
$$

其中 $y_w$ 为更受偏好的回答，$y_l$ 为较差的回答，$r_\phi$ 为奖励模型。

### 2.2 数据构建

#### 数据格式

每条数据包含一个 prompt 和至少两个回答的偏好排序。最佳实践是使用 **排名列表**（如 4 个回答的排序），而不是仅成对比较。

#### EOS Trick

损失仅在 EOS token 的位置计算。非 EOS token 的 logits 几乎总是负值，因此用一个常数 -1 奖励替代不完整的回答。这个技巧在 RM 训练和 PPO 中都很关键。

#### 数据质量关键

| 问题 | 解决方案 |
|------|---------|
| 标注员间不一致 | 多数投票 + 标注质量审核 |
| 回答差异不明显 | 仅保留 "significantly better" 或 "better" 的偏好对（Llama 3.1 实践） |
| 位置偏见 | 随机 shuffle 回答顺序 |
| 奖励黑客 | 使用多个 RM 集成 + 正则化 |

### 2.3 训练配置

#### 模型结构

- 在 SFT checkpoint 基础上增加一个随机初始化的线性 head（输出标量分数）
- 预训练模型权重冻结 + 仅训练 head，或全部微调

#### Llama 3.1 的 RM 训练实践

1. 每轮迭代 **从头训练** RM，使用全部累积的偏好数据
2. 去掉 margin loss，回归原始 ranking loss
3. 只使用 "significantly better" 或 "better" 的偏好对
4. 数据分布覆盖多个领域：数学、代码、创意写作、事实性问答等

#### DeepSeek-V3 的混合 RM 方案

| RM 类型 | 适用范围 | 方法 |
| ------ | ------- | --- |
| Rule-Based RM | 数学、代码 | 确定性验证（答案匹配 + 编译测试） |
| Model-Based RM | 开放性任务 | 训练时加入推理链以降低 reward hacking |

### 2.4 开源工具

## 3. RLHF 经典范式：PPO

### 3.1 原理

PPO（Proximal Policy Optimization）是 RLHF 的核心算法。在 RM 提供奖励信号后，PPO 优化策略模型以最大化奖励，同时通过 KL 散度惩罚防止策略与 SFT 基座偏离过远。

**PPO 目标函数**：
$$
\mathcal{L}_{\text{PPO}} = \mathbb{E}_{x,y \sim \pi_{\theta_{\text{old}}}} \left[ \min \left( \frac{\pi_\theta(y|x)}{\pi_{\theta_{\text{old}}}(y|x)} A, \text{clip}\left(\frac{\pi_\theta}{\pi_{\theta_{\text{old}}}}, 1-\epsilon, 1+\epsilon\right) A \right) - \beta \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}}) \right]
$$

### 3.2 四阶段流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  Pretrained   │───→│    SFT       │───→│  Reward Model │───→│  PPO RL      │
│  Model        │    │ (Demo Data)  │    │  (Rank Data)  │    │ (RM + KL)    │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

1. **SFT**：在有监督的演示数据上微调基座模型
2. **RM 训练**：在偏好排名数据上训练奖励模型
3. **PPO**：使用 PPO 算法优化策略，最大化 RM 奖励 + KL 惩罚
4. **迭代**：重复上述过程，每次使用当前最佳模型采样新数据

### 3.3 关键实现细节（OpenAI 的 25 项技巧）

| 技巧 | 说明 |
|------|------|
| **EOS Trick** | 非 EOS token 用 -1 常数奖励替代 |
| **Reward Whitening** | 略微降低胜率，缩短回答长度，但更稳定 |
| **KL 自适应** | 根据 KL 散度大小动态调整 KL 惩罚系数 |
| **Value Model 初始化** | 用 RM 初始化 PPO 的 Value Model（TÜLU 3） |
| **梯度检查点** | 节省显存的关键技术 |
| **Pre-rollout** | 在 PPO 更新前先采样一批回答计算基线 |

### 3.4 显存占用

| 组件 | 说明 | 显存占比 |
|------|------|---------|
| 策略模型（Policy） | 需推理 + 训练 | ~40% |
| 参考模型（Reference） | 只推理，计算 KL | ~20% |
| 奖励模型（RM） | 只推理，提供奖励 | ~20% |
| 价值模型（Value Head） | 附加在 Policy 或独立 | ~10% |
| 优化器状态 | Adam 动量 + 方差 | 剩余 |

**结论**：PPO 需要同时加载 4 个模型，是 DPO 的 2 倍显存消耗。

### 3.5 PPO 训练流程（TRL）

```python
from trl import PPOConfig, PPOTrainer, AutoModelForCausalLMWithValueHead
from transformers import AutoTokenizer
import torch

model = AutoModelForCausalLMWithValueHead.from_pretrained("gpt2")
tokenizer = AutoTokenizer.from_pretrained("gpt2")
config = PPOConfig(model_name="gpt2", learning_rate=1.41e-5)

ppo_trainer = PPOTrainer(config=config, model=model, tokenizer=tokenizer)

query = "Explain quantum computing"
input_ids = tokenizer(query, return_tensors="pt").input_ids
response = model.generate(input_ids, max_length=100)
reward = torch.tensor([0.8])  # 来自 RM 的评分

ppo_trainer.step([input_ids[0]], [response[0]], reward)
```

### 3.6 PPO 的困境

| 问题 | 表现 | 原因 |
|------|------|------|
| Reward Hacking | 模型学会欺骗 RM | RM 无法覆盖所有边界情况 |
| KL 崩塌 | 策略多样性急剧下降 | KL 惩罚不足或学习率过高 |
| 训练不稳定 | Loss 剧烈波动 | 优势估计不准确 |
| 计算开销大 | 需 4 模型同时加载 | 成本是 DPO 的 2~3 倍 |

> **2025 年趋势**：大厂（OpenAI、Anthropic）仍以 PPO 为核心，但中小团队正加速转向 DPO/GRPO。

## 4. 直接偏好优化：DPO 及其变体

### 4.1 DPO 原理

DPO（Direct Preference Optimization）绕过了显式的 RM 训练和 PPO 强化学习，直接从偏好对优化策略。核心洞察是：奖励函数可以表示为策略的闭式解。

**DPO 损失函数**：
$$
\mathcal{L}_{\text{DPO}} = -\mathbb{E}_{(x,y_w,y_l) \sim \mathcal{D}} \left[ \log \sigma \left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right) \right]
$$

**优点**：
- 无需 RM 训练——简化管线
- 无需 PPO 强化学习——训练稳定（像 SFT 一样收敛）
- 显存需求仅为策略 + 参考 2 个模型

**缺点**：
- 仅在偏好数据分布上优化，泛化能力较 PPO 弱
- 对偏好数据质量高度敏感
- On-policy 数据优于 off-policy 数据

### 4.2 数据格式与准备

```jsonl
# JSONL 格式，每行一个 JSON 对象
{"instruction": "写一首关于春天的诗", "output": [{"content": "春眠不觉晓..."}, {"content": "不知道怎么写诗"}]}
```

关键：chosen 和 rejected 的**优劣差异必须明显**。模糊的偏好对会给 DPO 带来噪声。

**数据注册**（LLaMA-Factory）：
```yaml
dataset:
  - dataset_name: my_preference_data
    ranking: true   # 标记为偏好数据集
```

### 4.3 DPO 训练配置

```python
from trl import DPOConfig, DPOTrainer
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")
tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-0.5B-Instruct")

training_args = DPOConfig(
    output_dir="Qwen-DPO",
    beta=0.1,                    # KL 惩罚系数，核心超参数
    learning_rate=5e-6,          # 比 SFT 小
    max_length=2048,
    max_prompt_length=1024,
)
trainer = DPOTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,       # 需要 chosen/rejected 字段
    processing_class=tokenizer,
)
trainer.train()
```

#### 调参指南

| 参数 | 作用 | 调优策略 |
|------|------|---------|
| `beta` | KL 惩罚系数（推荐 0.1~0.5） | 通用能力下降则**调高**，偏好对齐不足则**调低** |
| `learning_rate` | 学习率（推荐 1e-6~1e-5） | 比 SFT 低 2~10 倍 |
| `rpo_alpha` | DPO+SFT 混合权重（TRL v0.20） | 设置为 1.0 时同时优化 SFT loss |
| `loss_type` | 损失函数形式（sigmoid/ipo/kto_pair） | 默认 sigmoid 最稳定 |

#### 训练监控指标

| 指标 | 正常范围 | 说明 |
|------|---------|------|
| `rewards/chosen` | 增长 | 模型给 chosen 回答的打分 |
| `rewards/rejected` | 下降 | 模型给 rejected 回答的打分 |
| `rewards/accuracies` | 70%~90% | 偏好分类准确率 |
| `rewards/margins` | >0 且递增 | chosen 与 rejected 的分数差距 |

### 4.4 DPO 变体对比

| 方法 | 年份 | 核心创新 | 优势 | 适用场景 |
|------|------|---------|------|---------|
| **DPO** | 2023 | 闭式奖励替代 PPO | 稳定、简单 | 通用偏好对齐 |
| **KTO** | 2024 | 二值反馈（点赞/点踩） | 只需单个反馈，而非偏好对 | 数据稀疏场景 |
| **ORPO** | 2024 | Odds Ratio 合并 SFT+DPO | 零外部依赖，消除分布偏移 | 从头对齐（无参考模型） |
| **SimPO** | 2024 | 平均 log 概率作为隐式奖励 | 移除参考模型 | 节省显存（仅 1 模型） |
| **EXPO** | 2025 | Explicit Preference Optimization | 避免 DPO 重参数化缺陷 | 高品质偏好数据 |
| **MPO** | 2025 | 混合损失（DPO+BCO+SFT） | 兼顾对齐+能力保持 | 全面对齐场景 |

```bash
# KTO
```

### 4.5 统一理论框架

2025 年的研究（Wang et al., 2025）证明了 SFT 和 DPO 实际上在同一最优策略-奖励子空间中操作。这意味着：

- **SFT ≈ DPO + 弱 KL 约束**
- 两者可以统一建模为 **正则化最大似然估计**
- 在实践中可考虑 SFT 后直接接 DPO，跳过 RM 阶段

## 5. 新兴后训练方法

### 5.1 GRPO（Group Relative Policy Optimization）

#### 核心创新

由 DeepSeek 提出，消除了 PPO 中的 Critic（Value Model），通过对每个 prompt 采样 **一组回答**（通常 8~64 个），用组内平均奖励作为基线。

**优势估计**：
$$
A_i = \frac{r_i - \text{mean}(r_1, \ldots, r_G)}{\text{std}(r_1, \ldots, r_G)}
$$

**GRPO 目标**：
$$
\mathcal{L}_{\text{GRPO}} = \frac{1}{G} \sum_{i=1}^G \left[ \min \left( \frac{\pi_\theta(o_i|q)}{\pi_{\theta_{\text{old}}}(o_i|q)} A_i, \text{clip}\left(\frac{\pi_\theta(o_i|q)}{\pi_{\theta_{\text{old}}}(o_i|q)}, 1-\epsilon, 1+\epsilon\right) A_i \right) - \beta \cdot \text{KL}(\pi_\theta \| \pi_{\text{ref}}) \right]
$$

#### 优势

| 维度 | PPO | GRPO |
|------|-----|------|
| Critic 模型 | 必需 | **不需要** |
| 显存占用 | 4 个模型 | 2 个模型 |
| 奖励源 | RM + KL | 任意奖励函数 |
| 长推理任务 | 不稳定 | **优秀** |
| 实现复杂度 | 高 | 中等 |

#### 奖励函数设计（实战经验）

```python
def grpo_reward_func(completions, answers, **kwargs):
    """GRPO 奖励函数：格式 + 准确度"""
    rewards = []
    for completion, answer in zip(completions, answers):
        # 格式奖励
        has_think = 1 if "思考过程" in completion else 0
        # 准确度奖励
        is_correct = 1 if extract_answer(completion) == str(answer) else 0
        # 综合
        rewards.append(0.5 * has_think + 0.5 * is_correct)
    return rewards
```

**二阶段策略**：
1. **阶段一**：平滑奖励（如 0~1 连续值），让模型先学习基本模式
2. **阶段二**：严格 0/1 奖励，对齐真实业务目标

#### TRL GRPOTrainer 示例

```python
from trl import GRPOTrainer, GRPOConfig

def reward_func(completions, **kwargs):
    """奖励函数：返回列表"""
    return [float(len(set(c))) for c in completions]  # 示例：唯一字符数

trainer = GRPOTrainer(
    model="Qwen/Qwen2-0.5B-Instruct",
    reward_funcs=reward_func,
    train_dataset=dataset,
    args=GRPOConfig(
        max_steps=200,
        learning_rate=3e-4,
        beta=0.04,
        entropy_filtering=True,       # v0.20 新特性
        use_transformers_paged=True,  # 分页注意力
    ),
)
trainer.train()
```

### 5.2 DAPO（Dynamic Advantage Policy Optimization）

由字节跳动 + 清华大学于 2025 年提出，专门针对长链推理（CoT）训练的不稳定性问题。

**四大关键技术**：

| 技术 | 解决的问题 | 方法 |
|------|-----------|------|
| Clip-Higher | 熵崩塌 | 对正优势使用更高 clip 上限 |
| Dynamic Sampling | 无信息样本 | 过滤全部正确或全部错误的 prompt |
| Token-level Loss | 长序列梯度消失 | 逐 token 计算损失而非序列级 |
| Overlong Reward Shaping | 截断奖励噪声 | 处理超过最大长度的样本 |

**效果**：Qwen2.5-32B 在 AIME 2024 达到 50 分，训练步数比 DeepSeek-R1-Zero 少 50%。

### 5.3 RLVR（Reinforcement Learning with Verifiable Rewards）

**核心洞察**：数学、代码等任务不需要人类判断质量——单元测试、数学验证器即可提供奖励信号。

```
RLVR 流程：
问题 → 模型采样 → 多个回答 → 验证器检查（Pass/Fail） → 策略更新
```

#### DeepSeek-R1 的发现

- 纯 RLVR 训练产生了**涌现能力**：自我反思、动态策略切换、CoT 自发生成
- 这是纯 RL 在无人类监督数据下催生的高级行为

#### TÜLU 3 的 RLVR 实践

- 使用 PPO 算法进行 RLVR
- **关键发现**：不应混用 verifiable reward 和 RM score（会引入噪声）
- 用 General RM 初始化 PPO 的 Value Model

#### RLVR 奖励设计

| 任务类型 | 验证器 | 奖励值 |
|---------|--------|--------|
| 数学 | 答案精确匹配 | 0/1 |
| 代码 | 编译测试 + 单元测试 | 0/1 或部分分 |
| 逻辑推理 | 规则验证 | 0/1 |
| 事实性问答 | 检索验证 | 0~1 (连续) |

### 5.4 Constitutional AI（Constitutional AI / RLAIF）

**两阶段流程**：

1. **监督阶段**：模型使用宪法原则对自身输出进行自我批评和修订
2. **RL 阶段**：使用 AI 反馈（RLAIF）替代人类反馈——AI 模型标记偏好

**优点**：
- 仅宪法由人类编写，所有标注和批评由 AI 驱动
- 可扩展到不同文化和价值观体系
- 开源框架（Anthropic 提出）被 Claude 采用

### 5.5 Rejection Sampling + 迭代训练

#### RAFT（Rejection Sampling Fine-Tuning）

2025 年 Salesforce 的研究表明，简单的拒绝采样微调与 GRPO、PPO 竞争力相当。

```
流程：采样 → 筛选高分回答 → 仅在高分上做 SFT
```

**RAFT++** 的改进：
- 加入重要性采样
- 加入 Clipping 机制

#### Reinforce-Rej

过滤掉"全部正确"或"全部错误"的 prompt，在 KL 效率和稳定性上优于 GRPO。

```
核心发现：GRPO 的关键成功因素不是组归一化本身，
而是丢弃"全部错误"的 prompt（来自 Minimal-RL 研究）
```

#### 迭代式 Rejection Sampling SFT

```
循环：
1. 从当前策略采样多个回答
2. 用 RM 评分，接受概率 P_accept ∝ exp((r - r_max)/β)
3. 在接受的回答上做 SFT
4. 更新策略 → 回到 1
```

### 5.6 GSPO（Group Sequence Policy Optimization）

2025 年 7 月由 Qwen 团队提出，作为 GRPO 的替代方案。

**GRPO 的固有问题**：
- Token-level 重要性采样在长序列中方差累积严重
- 对 MoE 模型尤其有挑战：约 10% 的激活专家在每次梯度更新后变化
- 需要 Routing Replay 等辅助技巧才能收敛

**GSPO 的解决方案**：
- 重要性采样从 token 级 → **序列级**
- 应用长度归一化防止长度偏见
- 无需 Routing Replay 等辅助技巧
- 更好的可扩展性

## 7. 主流模型后训练工程实践

本章深入剖析当前六大主流大模型（Llama 3.1、DeepSeek-R1/V3、Qwen3、TÜLU 3、GPT-4o、Claude）的后训练管线设计，展示它们如何有机结合前述各方法。

### 7.1 Llama 3.1（Meta）

**管线概览**：
```
Base Model → Rejection Sampling → SFT → DPO（迭代 6 轮）
                ↑                      │
                └── 当前最佳模型采样 ────┘
```

#### 详细流程

**第 1 步：数据构建与清洗**
- 从多个来源收集数据（公开数据集 + 合成数据 + 人工标注）
- 四维评分：主题分类、质量（1-5）、难度、语义去重（MinHash + LSH）
- 高质量数据多 epoch，低质量数据降采样

**第 2 步：迭代式 Rejection Sampling（K=10~30）**
- 对每个 prompt，当前最佳模型采样 10~30 个回答
- 用 RM 评分，选最高分作为 SFT 训练数据
- 每轮迭代全部重新采样（确保 on-policy）

**第 3 步：SFT**
- 在 rejection sampling 筛选出的数据上做标准 SFT
- 数据量随迭代增长（从数千到数万）

**第 4 步：DPO**
- 偏好数据来源：前几轮迭代中表现最好的模型采样
- 每次只取最新批次
- **配置**：beta=0.1，SFT loss weight=0.2（同时优化 DPO + SFT）
- 使用 length-normalized DPO（优于标准 DPO 和 SimPO）

**第 5 步：RM 重训**
- 每轮迭代**从头训练**RM
- 使用全部累积的 preference data
- 移除 margin loss，回归原始 ranking loss
- 只使用"明显更好"或"更好"的偏好对

**工程要点**：
- 6 轮迭代，每轮数据管道全量重跑
- DPO 的 on-policy 数据比 off-policy 效果好
- 增加 prompt 多样性比重复使用同一 prompt 更有效

---

### 7.2 DeepSeek-R1 / V3

**管线概览**：
```
Base → Cold Start SFT (1K CoT) → GRPO (Reasoning) → Rejection Sampling + SFT → Comprehensive RL
```

这是目前**最完整的后训练管线**之一，清晰展示了 GRPO 如何与 SFT 和传统 RLHF 配合。

#### 阶段一：冷启动 SFT

| 项目 | 数据 |
|------|------|
| 数据量 | ~1,000 条高质量长 CoT 示例 |
| 数据来源 | DeepSeek-R1-Zero 采样 + 人工精修 |
| 格式 | `<think>推理过程</think><answer>答案</answer>` |
| 目的 | 建立基础推理格式，防止 RL 阶段早期探索混乱 |

**为什么需要冷启动**：
- DeepSeek-R1-Zero（纯 RL，无 SFT）出现语言混杂（language mixing）、格式不一致问题
- 少量高质量 SFT 数据提供了"方向性"引导

#### 阶段二：推理导向 GRPO

| 参数 | 数值 |
|------|------|
| Group size | 8~64 个采样 / prompt |
| 奖励信号 | 准确性 + 格式 + 语言一致性 |
| 训练步数 | 数万步（逐步增加难度） |
| 基座模型 | DeepSeek-V3-Base（671B MoE） |

**奖励设计**（三条腿走路）：
```
Reward = α × 准确性 + β × 格式 + γ × 语言一致性
         (规则匹配)      (标签检测)      (语言分类器)
```

**涌现行为**：模型自行学会了自我反思、纠错、长链推理——即著名的"Aha moment"。

#### 阶段三：拒绝采样 + SFT 蒸馏

```
步骤：
1. 用阶段二的 RL checkpoint 采样大量回答
2. 保留正确 + 格式良好的回答（rejection sampling）
3. 混合非推理数据（600K 推理 + 200K 通用）
4. SFT 1~2 epoch
```

**目的**：防止"过度适应推理"，保持通用语言能力。

#### 阶段四：综合 RL（对齐）

| 领域 | 奖励信号 | 方法 |
|------|---------|------|
| 数学、代码 | 规则奖励（确定性） | 继续 GRPO |
| 开放问答 | 人类偏好 RM | PPO-style 更新 |
| 安全性 | 安全规则 + RM | 约束优化 |

#### DeepSeek 工程要点总结

| 阶段 | 方法 | 数据量 | 目的 |
|------|------|--------|------|
| 冷启动 | SFT | ~1K | 格式引导 |
| 推理 RL | GRPO | 无限（自采样） | 推理爆发 |
| 蒸馏 | Rejection Sampling + SFT | ~800K | 泛化保持 |
| 综合 RL | GRPO + RM | 混合 | 全场景对齐 |

### 7.3 Qwen3（阿里巴巴）

**管线概览**（大模型：32B / 235B）：
```
Cold Start SFT (Long-CoT) → GRPO (Reasoning RL) → SFT (Mode Fusion) → General RL (20+ tasks)
```

Qwen3 的核心创新是**思考模式（Thinking）与非思考模式（Non-Thinking）融合于单一模型**。

#### 阶段一：长 CoT 冷启动 SFT

- 用高质量监督数据微调
- **查询过滤**：去除过于简单和难以验证的问题
- **回答过滤**：确保具备基础推理能力（不强求高推理强度）

#### 阶段二：推理 RL（GRPO）

- 使用 GRPO 增强推理能力
- 熵控制：平衡探索与利用
- 奖励信号：规则验证（数学、代码）+ 模型评分

#### 阶段三：思考模式融合 SFT（最关键创新）

```
┌──────────────────────────────────────────────────┐
│  Qwen3 统一模型                                   │
│                                                   │
│  <think>...</think> → [回答]  ← 思考模式          │
│  无 think 标签      → [回答]  ← 非思考模式        │
│  <auto>...</auto>   → 自适应选择                  │
└──────────────────────────────────────────────────┘
```

通过 SFT 将两种模式融合到同一模型中，用户通过 prompt 前缀控制模式。

#### 阶段四：通用 RL

覆盖 **20+ 任务类型**，使用三种奖励信号：

| 奖励类型 | 适用任务 | 说明 |
|---------|---------|------|
| 规则奖励 | 数学、代码 | 精确答案匹配 |
| 参考奖励 | 翻译、摘要 | 需标注答案的模型评分 |
| 无参考奖励 | 创意写作、开放问答 | 仅有 scoring rubric |

#### 小模型方案（≤30B）：蒸馏

小模型不跑完整的 4 阶段 RL，而是两阶段蒸馏：

1. **Off-policy 蒸馏**：合并教师模型（Qwen3-32B / 235B）的输出
2. **On-policy 蒸馏**：学生自己生成 → 用 KL 散度对齐教师分布

### 7.4 TÜLU 3（Allen AI）

**管线概览**：
```
SFT → DPO → RLVR (PPO-based) + General RM
```

TÜLU 3 是基于 Llama 3.1 的完全开源后训练研究，胜在**实验发现的价值**。

#### 阶段一：SFT

- 混合开源数据（WildChat、OpenAssistant、FLAN v2）
- **Sum Loss**：用 sum 而非 mean 处理 padding tokens（对变长序列更公平）

#### 阶段二：DPO

- 使用 **length-normalized DPO**
- 实验表明优于标准 DPO 和 SimPO
- On-policy 数据优于 off-policy 数据

#### 阶段三：RLVR（基于 PPO）

**关键设计**：
- 使用 **General RM 初始化 Value Model**
- **关键发现**：不要混用 verifiable reward 和 RM score！两者会互相干扰
- 数学问题用规则验证，代码用编译测试

#### TÜLU 3 的实验发现总结

| 发现 | 实践建议 |
|------|---------|
| DPO 中 on-policy 数据更优 | 用模型自身采样而非固定数据集 |
| RM score + verifiable reward 不兼容 | 两者分开使用，不要混入同一训练 |
| RLVR 提升推理但可能降低通用能力 | 需要混合通用奖励来平衡 |
| Prompt 多样性 > 回答多样性 | 增加 prompt 数量比更换回答对更有效 |

### 7.5 GPT-4o / GPT-4 系列（OpenAI）

**管线概览**：
```
SFT (Demo Data) → RM (Human Rankings) → PPO (RLHF) + RBRM (Safety)
                                                    ↑CriticGPT (质量提升)
```

#### 标准 RLHF 流程

OpenAI 的基本方法仍是 SFT → RM → PPO 三阶段，但增加了两大创新：

**1. RBRM（Rule-Based Reward Models, 2024）**

不是替代 RM，而是作为 PPO 中的 **额外奖励信号**：

```python
# RBRM 工作方式
rbrm_score = GPT-4-zero-shot(
    prompt=user_input,
    output=policy_output,
    rubric="""
    (A) 正确拒绝
    (B) 错误拒绝
    (C) 包含不允内容
    (D) 安全非拒绝回答
    """
)
# RBRM 分类结果作为辅助奖励加入 PPO
final_reward = rm_score + λ * rbrm_score
```

**效果**：拒绝不当内容的倾向性提高 82%，敏感请求按策略正确处理率提高 29%。

**2. CriticGPT（2024）**

一个基于 GPT-4 训练的批评模型，用于替代人类标注员进行 RLHF 的质量评估：

```
训练方法：
1. 人类在 ChatGPT 生成的代码中故意植入错误
2. CriticGPT 学习找出这些错误
3. 形成对抗循环（类似 GAN）
```

**Forced Sampling Beam Search（FSBS）**：推理时策略，平衡"全面性"和"幻觉率"。
**结果**：CriticGPT 检测出 24% 原本被人类评为"完美"的样本存在问题。

#### OpenAI 后训练工程要点

| 组件 | 方法 | 2024-2025 演进 |
|------|------|---------------|
| 奖励信号 | RM + RBRM | 增加规则化引导 |
| 数据质量 | 人工 + CriticGPT | AI辅助质量评估 |
| 安全对齐 | RBRM + 约束优化 | 可编程安全规则 |
| 领域适配 | Reinforcement Fine-Tuning (RFT) | 2024年底推出，仅需12~50样本 |

#### Reinforcement Fine-Tuning（RFT）

- 基于 RL 的领域微调，与 o1 模型家族兼容
- 仅需 12~50 个样本即可显著提升
- 使用 dense reward（部分正确也给分）
- 适用：分类、结构化提取、科学计算

### 7.6 Claude 系列（Anthropic）

### 7.8 后训练管线设计通用原则

综合以上主流模型的后训练实践，可以提炼出以下通用设计原则：

#### 原则一：阶段化递进

```
基础能力 → 单任务优化 → 多任务泛化 → 全场景对齐
(SFT)      (GRPO/PPO)   (RS + SFT)   (综合 RL)
```

每个阶段聚焦不同目标，逐步递进。DeepSeek-R1 的 4 阶段和 Qwen3 的 4 阶段都是这一思路的体现。

#### 原则二：选择匹配场景的奖励信号

| 场景 | 推荐奖励 | 原因 |
|------|---------|------|
| 数学推理 | 规则奖励 | 确定性验证，无 reward hacking |
| 代码生成 | 编译测试 + 单元测试 | 自动验证 |
| 安全对齐 | 分类器 + 规则 | 可编程、可审计 |
| 创意写作 | Human-preference RM | 主观性强 |
| 翻译、摘要 | 参考答案评分 | 有标注标准 |

#### 原则三：迭代数据优于静态数据

- TÜLU 3：DPO 中 on-policy 数据显著优于 off-policy
- Llama 3.1：每轮全部重新采样，使用当前最佳模型
- DeepSeek-R1：RL 产出的冷启动数据驱动下一阶段

#### 原则四：能力保持与对齐的平衡

| 策略 | 方法 | 代表模型 |
|------|------|---------|
| 数据混合 | SFT 中加入通用数据 | DeepSeek-R1 (600K+200K) |
| 损失混合 | DPO + SFT 联合优化 | Llama 3.1 (α=0.2) |
| 多阶段 | 推理 RL → 综合 RL | Qwen3 (20+ tasks) |
| 知识蒸馏 | 管线的最后阶段 | Qwen3 小模型 |

### 7.9 工程实践建议

#### 起步管线（推荐大多数团队）

```
SFT (1K~10K 高质量指令) → DPO (5K~50K 偏好对) → 评估 → 迭代
```

这是 Llama 3.1 验证过的路径，成本可控、效果明确。

#### 进阶管线（需要推理能力）

```
SFT (冷启动 CoT) → GRPO (规则奖励) → Rejection Sampling → SFT (混合) → DPO (偏好对齐)
```

参考 DeepSeek-R1 和 Qwen3 的设计。

#### 全面管线（追求 SOTA）

```
第一层：基础 SFT（大规模精选数据）
第二层：领域 RT（GRPO + 规则奖励，每个领域独立）
第三层：偏好对齐（DPO 或 PPO + RM）
第四层：综合 RL（多任务 + 多奖励信号融合）
第五层：安全约束（规则 + 红队 + RBRM）
```

#### 工具推荐

| 需求 | 推荐 |
|------|------|
| 快速验证 | LLaMA-Factory WebUI |
| 标准后训练 | Hugging Face TRL |
| 大规模 RL 训练 | OpenRLHF / veRL |
| 长序列训练 | 360-LLaMA-Factory |
| 基于规则的 RL | EasyR1 |
| 推理模型训练 | DAPO (字节开源) |
