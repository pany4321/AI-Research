## 1. 大模型微调概述

### 1.1 什么是大模型微调

大模型微调（Fine-Tuning）是指在已预训练的大语言模型（LLM）基础上，使用特定领域或任务的数据进行额外训练，使模型适应特定应用场景的过程。预训练阶段模型学习了通用的语言知识和推理能力，微调阶段则将这些能力"定向"到目标领域。

### 1.2 为什么需要微调

| 需求 | 说明 |
|------|------|
| **领域适配** | 通用模型缺乏行业专业知识（医疗、法律、金融等） |
| **风格控制** | 需要模型以特定风格或格式输出 |
| **知识更新** | 模型训练数据有截止日期，需要注入新知识 |
| **指令遵循** | 提升模型对特定指令格式的响应能力 |
| **性能提升** | 在特定任务上超越通用模型的表现 |

### 1.3 微调 vs. RAG vs. Prompt Engineering

```
Prompt Engineering  →  零成本，但效果有限
         ↓
   RAG（检索增强）  →  中等成本，适合知识密集场景
         ↓
    Prompt Tuning   →  需少量训练，适合简单模板任务
         ↓
    LoRA/QLoRA微调  →  需一定算力，效果显著
         ↓
   全量参数微调     →  高成本，极致效果
         ↓
    RLHF/DPO对齐    →  对齐人类偏好，提升可用性
```

## 2. 微调核心技术原理

### 2.1 全量微调（Full Fine-Tuning）

全量微调更新模型的所有参数。以一个 7B 模型为例，需要更新约 70 亿个参数。

**数学表达**：
- 预训练权重：$W_0 \in \mathbb{R}^{d \times k}$
- 微调后权重：$W = W_0 + \Delta W$
- 所有 $\Delta W$ 参数参与梯度更新

**特点**：
- ✅ 效果最优，理论上能达到最好的任务适配
- ✅ 无信息损失
- ❌ 显存需求极高（7B 模型需要约 120GB VRAM）
- ❌ 训练时间长，成本高
- ❌ 每个任务需保存完整模型副本

### 2.2 参数高效微调（PEFT）

PEFT（Parameter-Efficient Fine-Tuning）通过冻结大部分预训练参数，仅训练少量新增或选中的参数，大幅降低计算和存储成本。

**核心思想**：预训练模型已具备强大的特征提取能力，微调只需在现有能力基础上做"微小修正"。

**PEFT 方法分类**：

```
PEFT
├── 增量式（新增模块）
│   ├── Adapter       → 插入瓶颈网络
│   └── Prefix Tuning → 插入前缀向量
├── 选择性（选取子集）
│   ├── BitFit        → 只训练 bias 项
│   └── Partial FT    → 只训练最后几层
└── 重参数化（低秩分解）
    ├── LoRA          → 低秩矩阵分解（最主流）
    ├── QLoRA         → LoRA + 4bit量化
    └── AdaLoRA       → 自适应秩分配
```

### 2.3 LoRA 低秩适配（Low-Rank Adaptation）

#### 2.3.1 核心思想

LoRA 由微软于 2021 年提出，核心发现是：**大模型在下游任务微调时的权重更新矩阵 $\Delta W$ 具有内在的低秩特性**。这意味着 $\Delta W$ 可以用两个更小的矩阵的乘积来近似。

#### 2.3.2 数学原理

假设原始权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$：

- **全量微调**：需要训练 $d \times k$ 个参数
- **LoRA 方法**：将更新分解为两个低秩矩阵：
  - $A \in \mathbb{R}^{d \times r}$（随机高斯初始化）
  - $B \in \mathbb{R}^{r \times k}$（零初始化）
  - 其中 $r \ll \min(d, k)$，称为**秩（rank）**

最终更新：$\Delta W = B \cdot A$

微调后的权重：$W_{\text{new}} = W_0 + \alpha \cdot \frac{\Delta W}{r}$

其中 $\alpha$（lora_alpha）控制微调力度，$\frac{\alpha}{r}$ 为缩放因子。

#### 2.3.3 参数效率示例

对一个 $4096 \times 4096$ 的矩阵（约 1680 万参数）：

| 方法 | 训练参数量 | 节约比例 |
|------|-----------|---------|
| 全量微调 | 16,777,216 | - |
| LoRA (r=8) | 65,536 | **99.6%** |
| LoRA (r=16) | 131,072 | **99.2%** |
| LoRA (r=64) | 524,288 | **96.9%** |

#### 2.3.4 前向与反向传播

```
前向传播: y = W₀x + BAx
          先计算 A·x（降维），再乘 B（升维），减少计算量

反向传播: 仅对 A、B 计算梯度
          W₀ 完全冻结，不参与更新

初始状态: B 初始为 0 → BAx = 0 → 模型行为与原始一致
          保证训练初期的稳定性
```

#### 2.3.5 推理阶段零延迟

训练完成后可将低秩矩阵合并到原始权重：

$$W_{\text{merged}} = W_0 + \frac{\alpha}{r} \cdot BA$$

推理时使用合并后的权重，**零额外计算延迟**。且可为基础模型训练多个适配器，任务间动态切换：

```
基础模型 W₀
  ├── + (A₁, B₁) → 问答任务
  ├── + (A₂, B₂) → 摘要任务
  └── + (A₃, B₃) → 代码生成
```

### 2.4 QLoRA 量化低秩适配

#### 2.4.1 三大关键技术

QLoRA = **4-bit 量化基座** + **全精度 LoRA 适配器**，在 LoRA 基础上将内存需求再降低 4~5 倍。

##### ① NF4（4-bit NormalFloat）量化

- 针对神经网络权重**高斯分布**特性定制的 4-bit 刻度表
- 权重密集区域（0 附近）刻度更密，稀疏区域刻度更粗
- 相比普通 INT4 均匀量化，同比特数下精度更高

```
普通 INT4: |--|--|--|--|--|--|--|--|--|--|--|--|--|--|--|--|
            均匀分布刻度，中间区域精度浪费

NF4:      |--|----|------|--------|---|------|----|--|
            按高斯密度分布，中间密、两边疏
```

##### ② 双重量化（Double Quantization）

- **第一级**：对每个 block 的权重做 4-bit 量化，产生量化常数 scale
- **第二级**：对 scale 集合再做一次 8-bit 量化（对量化参数的"元量化"）
- 进一步节省约 200MB 内存

##### ③ 分页优化器（Paged Optimizer）

- 利用 NVIDIA 统一内存技术
- GPU 显存不足时自动将优化器状态转移到 CPU RAM（类似虚拟内存）
- 防止 OOM（Out Of Memory）错误

#### 2.4.2 QLoRA 训练流程

```
前向传播：低比特 W₀ 反量化到 FP16 → 与 LoRA 分支浮点增量相加 → 计算
反向传播：仅对 LoRA 权重 (A、B) 求梯度，W₀ 不更新
参数存储：W₀ 以 4-bit NF4 存储，LoRA 参数以 FP16/BF16 存储
```

#### 2.4.3 LoRA vs QLoRA 性能对比

| 对比维度 | LoRA (BF16) | QLoRA (4-bit NF4) |
|---------|-------------|-------------------|
| 可训练参数 | 0.1% | 0.1% |
| GPU 内存（7B 模型） | ~24 GB | **~6 GB** |
| GPU 内存（13B 模型） | ~48 GB | **~12 GB** |
| GPU 内存（70B 模型） | ~160 GB | **~48 GB** |
| 训练速度 | 1.0x | 0.6x（量化开销） |
| 效果评分（相对全量） | 98-99% | 97-98% |

### 2.5 其他 PEFT 方法对比

| 方法 | 提出时间 | 核心思想 | 训练参数量 | 推理延迟 | 最佳场景 |
|------|---------|---------|-----------|---------|---------|
| **Prompt Tuning** | Google 2021 | 输入层附加可学习虚拟 token | ~0.001% | 无 | 简单分类任务 |
| **Prefix Tuning** | Stanford 2021 | 每层 Attention 插入前缀向量 | ~0.1% | 轻微 | 生成任务 |
| **P-Tuning v2** | 清华 2022 | 每层 Transformer 加可学习 prompt | ~0.1-1% | 轻微 | 复杂 NLU 任务 |
| **Adapter** | Houlsby 2019 | FFN 后插入瓶颈网络 | ~0.5-8% | +4-6% | 多任务部署 |
| **LoRA** | 微软 2021 | Attention 旁路低秩矩阵 | ~0.01-0.1% | **零延迟** | 通用首选 |
| **QLoRA** | 2023 | LoRA + 4-bit 量化 | ~0.01-0.1% | 零延迟 | 消费级 GPU 微调 |

**选型建议**：

```
简单文本分类 → Prompt Tuning（最简单）
复杂 NLU 任务 → P-Tuning v2 或 LoRA
文本生成任务 → Prefix Tuning 或 LoRA
多任务部署   → Adapter（模块化切换）
显存受限     → QLoRA（消费级 GPU）
通用首选     → LoRA（效果/效率/易用性的最佳平衡）
```

## 4. 数据集准备

数据集是微调成功的核心。在 2025-2026 年的实践中，**数据质量远重要于数据数量**（5K-10K 高质量样本往往优于 100K 低质量数据）。

### 4.1 Alpaca 格式

**适用场景**：单轮指令任务（摘要、翻译、分类、代码生成、问答）

**结构**：

```json
{
  "instruction": "任务指令（必填）",
  "input": "额外的上下文输入（可选）",
  "output": "期望的模型输出（必填）",
  "system": "系统提示词（可选）",
  "history": [
    ["第一轮问题", "第一轮回答"],
    ["第二轮问题", "第二轮回答"]
  ]
}
```

**完整示例**：

```json
{
  "instruction": "请将以下句子翻译成英文",
  "input": "人工智能正在改变世界",
  "output": "Artificial intelligence is changing the world."
}
```

**多轮对话示例**：

```json
{
  "instruction": "你是一个专业的医疗助手，请回答患者的健康咨询问题。",
  "input": "我最近总是头痛，特别是早上起床的时候。",
  "output": "持续性头痛可能由多种原因引起，包括紧张性头痛、睡眠不足、颈椎问题等。建议您：1) 保持规律作息；2) 记录头痛的频率和部位；3) 如果持续超过一周，请及时就医。",
  "history": [
    ["你好，我最近身体不太舒服", "您好，我是医疗助手，请告诉我您的具体症状，我会尽力为您提供参考建议。"]
  ]
}
```

### 4.3 数据集注册与配置

以 LLaMA-Factory 为例，将数据集放入 `data/` 目录后，需在 `data/dataset_info.json` 中注册：

```json
{
  "my_training_data": {
    "file_name": "my_data.json",
    "formatting": "alpaca",
    "columns": {
      "prompt": "instruction",
      "query": "input",
      "response": "output"
    }
  },
  "my_chat_data": {
    "file_name": "my_chat_data.json",
    "formatting": "sharegpt",
    "columns": {
      "messages": "conversations"
    }
  }
}
```

然后在训练配置或 Web UI 中选择 `dataset: my_training_data` 即可。

### 4.4 数据质量最佳实践

#### 数据清洗

```
原始数据 → 去重 → 长度过滤 → 语言检测 → PII脱敏 → 格式标准化 → 最终数据集
```

- **去重**：过滤 prompt-response 对之间的高 n-gram 重叠
- **长度过滤**：剔除过短（< 10 tokens）或过长（超出模型 max_seq_length）的样本
- **语言检测**：确保所有数据符合目标语言
- **PII 脱敏**：使用正则 + NER 模型检测并替换个人信息

#### 质量提升技巧

1. **多样性优先**：覆盖目标场景的各种变体，避免数据模式过于单一
2. **人工校验种子集**：先写 100-200 条高质量种子数据，再用 LLM 扩展
3. **嵌入可视化**：使用 sentence embedding + UMAP 观察数据集的语义分布，识别覆盖盲区
4. **弱到强过滤（Superfiltering）**：用小模型（1B-3B）对数据打分，保留高分样本
5. **通用数据混合**：混入 5-10% 的通用指令数据，防止灾难性遗忘

#### 灾难性遗忘的预防

在领域数据中混合通用指令数据是关键策略。推荐比例：

```
领域特定数据 : 通用指令数据 = 90% : 10%
```

通用数据可选自公开数据集（如 Alpaca-GPT4、ShareGPT），用于维持模型的通用推理和对话能力。

## 7. 超参数调优指南

### 7.1 LoRA 关键超参数

| 参数 | 推荐范围 | 说明 |
|------|---------|------|
| `r`（秩） | 8-128 | 简单任务 8-16，复杂任务 32-64，极复杂任务 128 |
| `lora_alpha` | =r 或 2r | 控制微调力度，值越大更新幅度越大 |
| `lora_dropout` | 0.05-0.1 | 防止过拟合，数据量少时可适当增加 |
| `target_modules` | q_proj, v_proj 或 all | 对 Attention + FFN 层全部应用效果更好 |

**秩的选择原则**：

```
简单分类任务     → r=8
指令遵循微调     → r=16
领域知识注入     → r=32
复杂推理任务     → r=64
极致效果追求     → r=128+
```

### 7.2 训练超参数

| 参数 | 推荐范围 | 说明 |
|------|---------|------|
| `learning_rate` | 1e-5 ~ 5e-4 | LoRA 通常用 2e-4，全量微调用 1e-5 |
| `num_train_epochs` | 3-10 | 取决于数据量，小数据多 epoch，大数据少 epoch |
| `per_device_batch_size` | 1-8 | 受显存限制，结合 gradient_accumulation 使用 |
| `gradient_accumulation_steps` | 4-16 | 模拟更大 batch size，总 batch = per_device × grad_acc |
| `warmup_ratio` | 0.03-0.1 | 学习率预热比例 |
| `lr_scheduler_type` | cosine | 余弦退火调度器效果最佳 |
| `weight_decay` | 0.01-0.1 | 防止过拟合 |
| `max_grad_norm` | 0.3-1.0 | 梯度裁剪，防止梯度爆炸 |

### 7.3 数据集相关参数

| 参数 | 推荐范围 | 说明 |
|------|---------|------|
| `cutoff_len` | 1024-8192 | 根据任务需求设定，长文本任务用更大值 |
| `val_size` | 0.05-0.1 | 验证集比例 |
| `max_samples` | -1（全部） | 调试时可设小值快速验证 |

### 7.4 学习率调优经验

```
LoRA 微调通常起始学习率: 2e-4
         ↓ 如果 loss 震荡
      降低到 1e-4 或 5e-5
         ↓ 如果 loss 下降过慢
      增加到 5e-4
         ↓
    使用 cosine 调度 + warmup
```

### 7.5 过拟合判断与应对

| 信号 | 表现 | 对策 |
|------|------|------|
| **过拟合** | 训练 loss 持续下降，验证 loss 开始上升 | 增加 dropout、减小 r、增大数据量 |
| **欠拟合** | 训练 loss 和验证 loss 均未有效下降 | 增大 r、增大学习率、增加 epoch |
| **训练不稳定** | loss 剧烈震荡 | 减小学习率、增加 warmup、梯度裁剪 |

## 8. 常见问题与解决方案

### 8.1 显存不足（OOM）

**现象**：训练过程中出现 `CUDA out of memory` 错误。

**解决方案**（按优先级排序）：

1. **开启梯度检查点**（gradient_checkpointing）：以约 15% 速度换 40% 显存
2. **减小 batch size**：从 2 减到 1
3. **增加 gradient_accumulation_steps**：补偿 batch size 减小的影响
4. **降低序列长度**：减小 cutoff_len
5. **使用 QLoRA 替代 LoRA**：4-bit 量化可减少 60%+ 显存
6. **启用分页优化器**（Paged Optimizer）：QLoRA 内置，自动处理显存溢出

```yaml
# 显存优化配置
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
gradient_checkpointing: true
optim: paged_adamw_8bit  # 分页优化器
```

### 8.2 Loss 不下降

| 原因 | 排查方法 | 解决方案 |
|------|---------|---------|
| 学习率过小 | 尝试增大 5-10x | 调高 lr 到 2e-4~5e-4 |
| 数据格式错误 | 检查 dataset_info.json 配置 | 确保列映射正确 |
| 模型与模板不匹配 | 检查 template 参数 | Qwen 用 qwen，LLaMA 用 llama |
| LoRA 未生效 | 检查 target_modules | 设置 `lora_target: all` |

### 8.3 模型输出质量差

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 输出内容重复 | 学习率过高或训练不足 | 降低 lr、增加 epoch |
| 输出与基础模型无区别 | LoRA 未生效 | 检查 target_modules、增大 r |
| 输出无关内容 | 数据集格式错误 | 检查 Alpaca/ShareGPT 字段映射 |
| 灾难性遗忘 | 通用能力下降 | 加入 10% 通用指令数据混合训练 |

### 8.4 数据格式常见错误

```
错误：{"instruction": "...", "response": "..."}  ← 字段名错误
修正：{"instruction": "...", "output": "..."}     ← 必须是 output 字段

错误：dataset_info.json 中漏注册
修正：在 data/dataset_info.json 添加配置

错误：JSON 格式错误（最后一行的逗号等）
修正：使用 jsonlint 验证格式
```

### 8.5 训练速度优化

```bash
# 1. 安装 Flash Attention（大幅加速）
pip install flash-attn --no-build-isolation

# 2. 使用 BF16 替代 FP16（更稳定且略有加速）
bf16: true

# 3. 增加 num_workers
preprocessing_num_workers: 16

# 4. 启用 torch.compile（需要 PyTorch 2.0+）
# 在 YAML 配置中添加：
torch_compile: true  # 实验性功能，可能不兼容部分模型
```

## 9. 总结与展望

### 9.1 技术总结

大模型微调技术已从"资源密集型"走向"人人可用"：

- **LoRA 是当前的最佳平衡点**——在效果、效率、易用性上均表现优异
- **QLoRA 使消费级 GPU 微调成为现实**——4-bit 量化让 7B-14B 模型可在 RTX 4090 上微调
- **数据质量远重要于数据数量**——5000 条精心标注的数据往往优于 10 万条噪声数据
- **开源工具链成熟**——LLaMA-Factory 的 Web UI、Unsloth 的极致性能、Axolotl 的企业级能力覆盖了从入门到生产的所有场景

### 9.2 趋势展望（2026）

| 趋势 | 说明 |
|------|------|
| **小型化** | 小模型（1-4B）+ 微调 = 大模型 80%+ 的效果，成本极低 |
| **偏好对齐** | SFT → DPO/KTO/GRPO 成为标准流程，RLHF 因复杂度高逐渐边缘化 |
| **端侧部署** | 微调后的模型通过 GGUF 量化部署到手机和边缘设备 |
| **自动化数据** | 大模型自动生成 + 过滤高质量训练数据，减少人工标注成本 |
| **多模态微调** | Qwen3-VL 等模型支持图文混合微调，应用场景大大扩展 |
| **强化学习** | GRPO（如 DeepSeek R1 使用的方法）在推理增强上展现巨大潜力 |
